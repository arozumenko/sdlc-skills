# Copilot context injection via hooks — how role memory & shared docs reach the agent

This documents how sdlc-skills delivers **per-role memory** and **shared project
docs** to **GitHub Copilot** — both the standalone **Copilot CLI** and **VS Code
Copilot Chat** — i.e. the Copilot equivalent of Claude Code's `@import`.

Everything below is **empirically verified** against Copilot CLI `1.0.x` and the
VS Code Copilot runtime. Where measured behavior diverges from the docs, the
measured behavior is stated and wins.

---

## 1. Why this exists

Claude Code's `@import` (e.g. `@.agents/memory/<role>/SOUL.md` in an `AGENT.md`)
inlines the referenced file into context at prompt-assembly time — runtime-resolved,
always present, uncapped. **Copilot has no `@import`:** a bare `@path` / markdown
link in an agent body or hook output is **not** expanded; the model only *might*
read it on demand (lazy, not guaranteed — measured 0/10 on a generic ask). To get
content reliably into context we must inject the **actual content** through a
channel Copilot honors.

There are exactly **two** such channels:

| Channel | Written by | Reaches | Size | Freshness |
|---|---|---|---|---|
| **Instruction-file shelf** — `.github/instructions/<name>.instructions.md` | a hook, at session start (`refresh_shared_instructions`) | system prompt | **uncapped** | **warm-cache** (a write lands on the *next* session) |
| **`additionalContext`** — hook stdout | a hook, at fire time (`build_capped_context`) | the model's turn context | **≤ ~10 KB** | current session |

No hook output field can write the system prompt directly (verified: `systemPrompt`,
`systemMessage`, `appendSystemPrompt`, `instructions`, `modifiedConfig`,
`hookSpecificOutput` — only `additionalContext` reaches the model). The shelf is the
*indirect* system-prompt channel: instruction files are read at startup, so a hook
that (re)writes them affects the **next** session — "warm cache". For project docs
that change rarely, one-session-stale is effectively current.

---

## 2. The single most important measured fact

**A Copilot CLI _dispatched sub-agent_ does NOT inherit the instruction-file shelf.**
It runs in an isolated context window; the CLI loads `copilot-instructions.md` /
`.github/instructions/*.instructions.md` / `AGENTS.md` / `~/.copilot/copilot-instructions.md`
only into the **top-level** session's system prompt. Proven by planting unique
sentinels in all four file types **and** in a frontmatter hook — the CLI main agent
saw them; a dispatched sub-agent saw none (and described its own context as just
"repository info, working directory, agent config"). Frontmatter `hooks:` are ignored
by the CLI entirely.

**VS Code is different:** a dispatched sub-agent *does* inherit the shelf (proven with
the same sentinel, `applyTo: "**/*"`).

This asymmetry drives the whole design — see the matrix in §4. The CLI sub-agent is
the **only** audience without the shelf, so it's the only one that needs shared docs
pushed through `additionalContext`.

## 3. The pieces

| File | Role |
|---|---|
| `.github/hooks/sdlc-skills.json` | workspace hook config (events Copilot fires) — generated by `bin/init.mjs` |
| `.github/hooks/sdlc-skills/run-hook.cmd` | cross-platform launcher (cmd/PowerShell ↔ bash polyglot) |
| `.github/hooks/sdlc-skills/session-start` | session hook: refresh the shelf + build the primary/main-agent `additionalContext` |
| `.github/hooks/sdlc-skills/agent-start` | sub-agent hook: build the dispatched agent's `additionalContext` |
| `.github/hooks/sdlc-skills/lib.sh` | all shared helpers (see §10) |
| `.github/hooks/sdlc-skills/config.sh` | **project tunables** (sourced by `lib.sh`); seeded once from `config.sh.example`, never clobbered by sync |
| `.github/instructions/<name>.instructions.md` | the generated shelf (per-machine, regenerated each session); one file per shared doc |
| `.github/agents/<role>.agent.md` → `hooks:` frontmatter | generated by `init` for **every** agent; VS Code **primary-agent** role-memory injection (see §6) |

## 4. Delivery matrix (per audience)

Two kinds of content — **role memory** (role-specific: persona + memory index) and
**shared docs** (project-wide: `profile`, `workflow`, `testing`, `team-comms`, …).

| Audience | Role memory | Shared docs |
|---|---|---|
| **CLI main** (primary) | `additionalContext` — `session-start` resolves the `--agent` name from `~/.copilot/session-state/<sid>/events.jsonl` | **instruction-file shelf** |
| **CLI sub-agent** (dispatched) | `additionalContext` (`agent-start` reads `agentName`/`agent_type` from payload) | **`additionalContext`** — inlined by priority up to the cap, overflow → MUST-READ read-list (no shelf for it) |
| **VS Code main** (primary) | `additionalContext` — **frontmatter `SessionStart` hook** (nothing else knows the role in time — see §6) | **instruction-file shelf** |
| **VS Code sub-agent** (dispatched) | `additionalContext` — workspace **`SubagentStart`** (PascalCase) | **instruction-file shelf** |

**Role memory is role-specific, so it can NEVER go on the shelf** (the shelf is global
— `applyTo: "**/*"` — and would leak one role's memory to every agent). Memory always
rides `additionalContext`.

So the only cell where shared docs ride `additionalContext` is the **CLI sub-agent**.
Everywhere else they come off the shelf.

## 5. `build_capped_context` — the `additionalContext` builder

Signature: `build_capped_context <project> <role> <instr_present> <cli_sub>`. Fill
order, never exceeding the cap:

1. **Role memory first.** Each file in `role_memory_files()` (default `SOUL.md
   snapshot.md MEMORY.md`; `SOUL.md` injected only under Copilot — Claude/Codex get
   the persona via `@import`) is inlined while it still fits; a file too big to fit
   becomes a **self-read pointer** (read its own `.agents/memory/<role>/…`).
2. **Shared docs — only for a CLI sub-agent** (`cli_sub` set). Inlined in
   `shared_doc_names()` / `$SDLC_SHARED_DOCS` **priority order** until the cap;
   whatever doesn't fit → MUST-READ read-list. This is the **only** place the
   `SDLC_SHARED_DOCS` order matters (the cap forces the choice; for every other
   audience the shelf delivers all docs, order-independent).
3. **Shelf-getters** (everyone else): no shared inline. A read-list is added **only**
   if `instr_present` is empty — a cold first run before the shelf exists.

**Cap mechanics.** The cap is `$SDLC_CTX_CAP` (default `10240`). Sizes are measured
**after JSON escaping** by `esc_byte_len` (raw bytes + count of `\ " \n \r \t`, which
each gain a backslash) — that's what the runtime measures. The budget reserves room
for the **worst-case** MUST-READ directive (every memory file + every shared doc,
via `list_role_memory_files` + `list_shared_files`), so inlined content can never
crowd the directive out and push the payload over the cap. (A *fixed* reserve was not
enough — a long role name + full overflow list exceeded it; the over-cap payload is
then dropped **whole** by the runtime.)

**Non-Copilot** (Claude Code / Codex / Cursor, i.e. neither `COPILOT_CLI` nor
`SDLC_VSCODE` set): the cap doesn't apply — memory + all shared docs are inlined
as-is.

`cli_sub` derivation: in `agent-start`, `COPILOT_CLI` set ⇒ a CLI sub-agent (the CLI
only ever runs `agent-start` for a dispatched sub-agent; it ignores frontmatter
hooks). `SDLC_VSCODE` agents inherit the shelf, so `cli_sub=""`. `session-start`
always passes `cli_sub=""` (the main agent is never a CLI sub-agent).

`instr_present`: `instructions_present()` checks whether the shelf files already
exist; `session-start` captures it **before** `refresh_shared_instructions` (re)writes
them, so it reflects whether *this* session loaded them at startup.

The MUST-READ directive (`build_read_directive`) is presence-aware ("read each in full
now — skip any already present in your context or instructions"), so a shelf-getter
that gets the cold-start read-list doesn't redundantly re-read once the shelf is warm.

## 6. The instruction-file shelf — `refresh_shared_instructions`

Called from `session-start` (fires in both runtimes; side-effect only — doesn't change
this session's `additionalContext`). It mirrors **each** shared doc to its own file:

- `.agents/<name>.md` → `.github/instructions/<name>.instructions.md`, frontmatter
  **`applyTo: "**/*"`** (so it loads for all files), one file **per doc** (not a single
  combined blob — the old combined `sdlc-shared.instructions.md` is actively removed).
- **Idempotent:** rewrites only when the content changed (no working-tree churn).
- **Clobber-guarded:** every generated file carries an `auto-generated by sdlc-skills`
  marker; a same-named file *without* the marker (human-authored) is never overwritten.
- **Orphan-sweep:** a generated file whose source doc left `$SDLC_SHARED_DOCS` (or was
  deleted) is removed.
- **Tracking is the user's call:** `init` does **not** modify the project `.gitignore`
  — it only prints a hint. The shelf is per-machine and session-regenerated (not
  source), so gitignoring `.github/instructions/*.instructions.md` is recommended to
  avoid churn, but it's left to the user (some may prefer to commit it so a fresh
  clone has the shelf on session #1 instead of warming up).

Reach (proven): the shelf loads into the system prompt of the **CLI main agent** and
**all VS Code agents (main and dispatched sub-agents)** — but **not** CLI sub-agents
(§2).

### Primary vs sub-agent (role memory)

Every installed agent can run as the **primary** (user-invoked) agent *or* be dispatched
as a **sub-agent** — the injection path differs by which, not by role:

- **Primary agent** (top-level, user-invoked — an orchestrator like `test-automation-lead`,
  the interactive `scout`, or any worker a user selects directly):
  - **VS Code:** a **frontmatter `SessionStart` hook** — generated by `init` for **every**
    agent — injects its role memory. It's mandatory because VS Code can't identify the
    active agent at `SessionStart`; the agent is recorded only in
    `workspaceStorage/<hash>/chatSessions/<sid>.jsonl` ~90 s later. The frontmatter
    file *is* that role, so it passes the name explicitly (`agent-start <role>`).
  - **CLI:** resolved from `~/.copilot/session-state` by `session-start` (the CLI primary
    IS identified via `--agent`). The CLI ignores frontmatter hooks, so the generated
    hook is a harmless no-op there.
- **Sub-agent** (dispatched): the frontmatter `SessionStart` hook does **not** fire —
  dispatch raises `SubagentStart`, a different event — so memory comes via the workspace
  `subagentStart`/`SubagentStart` → `agent-start`. The two paths stay disjoint, so an
  agent never double-injects.

The frontmatter hook's `env` sets `SDLC_HOOK_EVENT: "SessionStart"` — used by
`emit_subagent_context` to label the emitted event name. (It no longer means "skip
shared docs"; `build_capped_context` decides shared delivery from `cli_sub` /
`instr_present`. A VS Code primary is a shelf-getter, so it gets memory only.)

Example frontmatter hook (generated for every agent by `injectCopilotSessionStartHook`):

```yaml
hooks:
  SessionStart:
    - type: command
      command: '"./.github/hooks/sdlc-skills/run-hook.cmd" agent-start test-automation-lead'
      env: { SDLC_VSCODE: "1", SDLC_HOOK_EVENT: "SessionStart" }
      timeout: 10
```

## 7. Hook engines & event casing

Both runtimes scan `.github/hooks/*.json`. There are **two hook engines**:

| | Engine | Event casing | Output consumed |
|---|---|---|---|
| Workspace `.github/hooks/*.json` | Copilot CLI engine (used by **both** CLI and VS Code) | **camelCase** | top-level `additionalContext` (ignores `hookSpecificOutput`) |
| Agent **frontmatter** `hooks:` | VS Code agent-hooks (Preview) | **PascalCase** | `hookSpecificOutput.additionalContext` |

Measured casing quirks the generated `sdlc-skills.json` accounts for:

- **CLI** fires both camelCase and PascalCase workspace entries but consumes only the
  camelCase one's top-level `additionalContext` (PascalCase `hookSpecificOutput`
  ignored → no double-injection).
- **VS Code** workspace fires `sessionStart` as camelCase but `SubagentStart` as
  **PascalCase** — so we ship both `subagentStart` (CLI worker) and `SubagentStart`
  (VS Code worker).
- The **CLI does not run agent-frontmatter hooks at all** (verified across casing/field
  variants, and again here with a sub-agent frontmatter hook whose probe never ran).

`lib.sh` picks the emit shape from the env flag each entry sets: `COPILOT_CLI=1` →
top-level `additionalContext`; `SDLC_VSCODE=1` → `hookSpecificOutput`. Generated config:

```json
{
  "version": 1,
  "hooks": {
    "sessionStart":  [{ "type": "command", "bash": "\"./.github/hooks/sdlc-skills/run-hook.cmd\" session-start", "powershell": "...", "env": { "COPILOT_CLI": "1" }, "timeoutSec": 10 }],
    "subagentStart": [{ "type": "command", "bash": "\"./.github/hooks/sdlc-skills/run-hook.cmd\" agent-start", "powershell": "...", "env": { "COPILOT_CLI": "1" }, "timeoutSec": 10 }],
    "SubagentStart": [{ "type": "command", "command": "\"./.github/hooks/sdlc-skills/run-hook.cmd\" agent-start", "env": { "SDLC_VSCODE": "1" }, "timeout": 10 }]
  }
}
```

## 8. Configuration (`config.sh`)

Project tunables live in `.github/hooks/sdlc-skills/config.sh`, **sourced by `lib.sh`**
(via `BASH_SOURCE`, so it finds the file beside itself). It is project-specific, lives
with the hooks, and is **not** overwritten by a package sync (sync refreshes the hook
*scripts*, not `config.sh`). `init` seeds it once from `config.sh.example` and never
clobbers an existing one. Each knob uses `: "${VAR:=default}"`, so a real environment
value still wins.

| Knob | Default | Meaning |
|---|---|---|
| `SDLC_SHARED_DOCS` | `testing profile workflow conventions role-overrides team-comms` | the curated shared-doc set **and** the inline **priority order** for CLI sub-agents (only matters there) |
| `SDLC_ROLE_MEMORY_FILES` | `SOUL.md snapshot.md MEMORY.md` | which `.agents/memory/<role>/` files to inject |
| `SDLC_CTX_CAP` | `10240` | the `additionalContext` byte cap |

A listed doc/file that doesn't exist is simply skipped.

## 9. Hard constraints (measured)

1. **`additionalContext` is capped at ~10 KB.** An over-cap payload is dropped **in
   full** (8 KB injects; >10 KB drops). `build_capped_context` enforces this (post-JSON-
   escape, with a worst-case-directive reserve) and **never emits over-cap**.
2. **The system prompt is sealed before hooks fire.** Agent body + instruction files
   are read at startup; a hook (re)writing them affects only the *next* session — the
   shelf's warm-cache (one-session-stale by design, not a defect).
3. **VS Code can't resolve the active agent at `SessionStart`** (recorded ~90 s later;
   the global `session-store.db` `agent_name` is empty for chat sessions;
   `~/.copilot/session-state` is CLI-only) → the frontmatter hook is mandatory for VS
   Code **primary-agent** memory, so `init` bakes it into every agent.
4. **CLI sub-agents are context-isolated** — no instruction-file shelf, no frontmatter
   hooks (§2). Their only channel is the workspace `subagentStart` → `additionalContext`.

## 10. `lib.sh` helper inventory

- `read_stdin_payload` — bounded background `cat <&0` (~2 s cap). The CLI leaves the
  `subagentStart` stdin pipe open (no EOF); a plain `cat` would hang to the 10 s hook
  timeout and the hook would be killed with no output. Pure bash 3.2 (`read -t`
  discards partial input on 3.2, so it's unusable).
- `is_blank` — `case`-glob emptiness test. **Replaces `[ -z "${var//[…]/}" ]`**, whose
  global substitution is **O(n²) on bash 3.2** (~114 s on a 14 KB string → blew the
  hook timeout → silent no-injection).
- `byte_len` / `esc_byte_len` — raw byte length / post-JSON-escape length (raw + count
  of backslash-gaining chars), used for cap math.
- `shared_doc_names` — the curated `$SDLC_SHARED_DOCS` set (env or default).
- `role_memory_files` — the `$SDLC_ROLE_MEMORY_FILES` set with `SOUL.md` Copilot-gated.
- `collect_shared_context` / `collect_role_memory` — concatenate docs/memory (used on
  the non-Copilot inline-all path).
- `list_shared_files` / `list_role_memory_files` — project-root-relative path lists
  (used for read-lists and the worst-case-directive reserve).
- `build_read_directive` — the presence-aware MUST-READ block.
- `build_capped_context` — the `additionalContext` builder (§5).
- `refresh_shared_instructions` — the shelf writer (§6).
- `instructions_present` — is the shelf on disk (cold-start gate).
- `resolve_cli_session_agent` — reads `~/.copilot/session-state` for the `--agent` name.
- `emit_session_context` / `emit_subagent_context` — emit in the runtime's shape
  (`hookSpecificOutput` for VS Code/Codex/Claude; top-level for CLI; raw for Kiro).
- `has_subagent_hook` / `is_claude_code` / `is_codex` — runtime detection.

## 11. Cross-platform & bash 3.2

- **`run-hook.cmd`** is a cmd/bash polyglot. On Windows, cmd/PowerShell locate
  Git-for-Windows `bash.exe` (standard paths or `PATH`) and exec the script; if no bash
  is found it exits 0 (plugin works, just without injection). On macOS/Linux the unix
  portion runs directly. The config wires it via `bash`+`powershell` (CLI) and `command`
  (VS Code) so both OS families resolve it.
- **bash 3.2 (macOS default) shapes `lib.sh`:** no associative arrays; `${var//[…]/}`
  is O(n²) (→ `is_blank`/`case`); `read -t` loses partial input (→ `read_stdin_payload`
  backgrounds `cat <&0`); `BASH_SOURCE` is available (→ `config.sh` sourcing). Scripts
  avoid heredocs (bash 5.3 heredoc hang) and use `printf`.
- **Project root:** `PROJECT_DIR="${CLAUDE_PROJECT_DIR:-${COPILOT_PROJECT_DIR:-$PWD}}"`.
  The Copilot CLI sets **`COPILOT_PROJECT_DIR`** (not `CLAUDE_PROJECT_DIR`); using it is
  essential or a CLI run resolves `.agents/` against the wrong dir and injects nothing.
- **`~/.copilot/session-state`:** `$HOME/.copilot` on all platforms (Git Bash maps
  `$HOME`→`%USERPROFILE%`); `resolve_cli_session_agent` honors `COPILOT_CONFIG_DIR` /
  `COPILOT_HOME` as a forward-compatible override.

## 13. Summary

- **Two channels.** The **instruction-file shelf** (`refresh_shared_instructions` →
  `.github/instructions/<name>.instructions.md`, `applyTo: "**/*"`, warm-cache,
  per-machine/regenerated — gitignoring it is the user's call) carries **shared docs**
  to the CLI main agent and **all** VS Code agents.
  **`additionalContext`** (`build_capped_context`, ≤10 KB, current-session) carries
  **role memory** to everyone and — for the one audience without the shelf, the **CLI
  sub-agent** — shared docs too (inlined by `SDLC_SHARED_DOCS` priority, overflow →
  MUST-READ read-list).
- **Role memory is role-specific** → never on the global shelf → always
  `additionalContext`.
- **CLI sub-agents don't get the shelf or frontmatter hooks; VS Code sub-agents do** —
  the asymmetry the whole design turns on.
- **`@path` references don't auto-inline** in Copilot — inject content, not references.
- **Config** is `config.sh` (seeded from `config.sh.example`, sourced by `lib.sh`).
