# Mobile Locator Strategy Reference

## Playwright Mode (PWA / Hybrid)

Standard web locators apply. Prefer in order:

1. `data-testid` → `data-qa` → `data-cy` (test IDs)
2. ARIA role + name: `role=button[name="Sign In"]`
3. Visible text: `text=Sign In`
4. `name` attribute (form fields)
5. CSS class (stable, non-generated only)
6. XPath (last resort)

**Mobile-specific note:** Touch targets on mobile are often larger wrappers around text. If a text locator fails, try targeting the parent `<button>` or `<a>` with the same accessible name.

---

## Manual Mode (Native)

In the execution guide, describe elements by their **visual label** and **location** so a human can find them without knowing selectors:

| ✅ Good | ❌ Bad |
|--------|--------|
| "the blue 'Sign In' button at the bottom of the screen" | "the submit button" |
| "the search field in the top navigation bar" | "the input" |
| "the ☰ hamburger menu icon (top-left corner)" | "the menu" |

## Appium Mode (Future Reference)

Priority order for native app locators:

### iOS (XCUITest)
1. **Accessibility ID** — `driver.findElement(By.AccessibilityId("SignInButton"))` — set via `accessibilityIdentifier` in code
2. **Predicate String** — `type == 'XCUIElementTypeButton' AND label == 'Sign In'`
3. **Class Chain** — `**/XCUIElementTypeButton[\`label == 'Sign In'\`]`
4. **XPath** — `//XCUIElementTypeButton[@label='Sign In']` — slowest, avoid if possible

### Android (UiAutomator2)
1. **Accessibility ID** — `driver.findElement(By.AccessibilityId("sign_in_btn"))` — set via `contentDescription` in code
2. **ID** — `driver.findElement(By.id("com.example.app:id/sign_in_button"))` — resource ID
3. **UiAutomator** — `new UiSelector().text("Sign In").className("android.widget.Button")`
4. **XPath** — `//android.widget.Button[@text='Sign In']` — avoid

### Cross-Platform Strategy

When authoring test cases for `platform: both`, prefer **Accessibility ID** — the same logical name can be used on both platforms if developers set it consistently (`accessibilityIdentifier` on iOS, `contentDescription` on Android).

```
# TC Step (platform-agnostic)
Tap the "Sign In" button  →  Accessibility ID: "SignInButton"
```

Document the Accessibility ID in `.agents/mobile-qa/app_profile.md` under **Reliable Locators** so all agents can reference it.

## Fragile Locator Patterns to Avoid

| Pattern | Why it breaks |
|---------|--------------|
| XPath with positional index (`//button[2]`) | Order changes with UI updates |
| Dynamic resource IDs (`com.example:id/btn_12345`) | Generated at runtime |
| XPath with text content that gets translated | Fails on non-English locales |
| CSS class names with hash suffixes (`btn-a1b2c3`) | Generated by CSS-in-JS bundlers |
| Coordinate-based taps (`tapAt(200, 400)`) | Breaks on different screen sizes |
