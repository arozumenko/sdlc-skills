# The Traceability Matrix

The matrix is the artifact Phase 1 (triangulation) produces. It makes the
requirement ↔ case ↔ result relation visible so the four gap classes can be
read directly off it. This file defines the layout, the column semantics, the
build-stamp rule, and a worked example that surfaces every gap class.

The matrix is **manual-discipline output** — it is built by reading
requirements, case files, and run reports, not generated by an automation
framework. Write it to `/tmp` (e.g. `/tmp/traceability-<epic>-<date>.md`) or
present it inline; it is a content layer over the consumer's tree, not a file
that litters their repo.

## Header — the build stamp is mandatory

Every matrix starts with the build it was triangulated against. A matrix
without a build stamp is itself stale — the reader can't tell whether any green
is trustworthy.

```markdown
# Traceability Matrix — {epic / requirement set}
- **Build under audit:** {version / commit / deploy tag}
- **Date triangulated:** {YYYY-MM-DD}
- **Sources:** requirements ({tracker / xlsx}), cases ({path / TMS}), results ({run report / TMS})
- **Coverage:** {covered}/{total} requirements trustworthy-green · {uncovered} uncovered · {orphans} orphan cases
```

## The grid

One **row per requirement**. The cells trace that requirement out to its cases
and each case's freshest result. Requirements that no case reaches still get a
row (with an empty Cases cell) — that empty row *is* the uncovered-requirement
finding made visible.

| Column | Meaning | How it surfaces a gap |
|---|---|---|
| **Req ID** | Stable requirement / story ID | The row key; an ID with no row means the requirement set wasn't fully ingested |
| **Req priority** | p0/p1 = must-work / critical-path; p2/p3 otherwise | Sets the default band for an uncovered-requirement or stale-result finding |
| **Requirement (testable form)** | The triaged, testable statement (post-Phase-0) | If this still reads ambiguous, Phase 0 wasn't run — stop and triage |
| **Case(s)** | Case IDs tracing to this requirement (comma-separated) | **Empty → uncovered-requirement** |
| **Result (freshest)** | PASS / FAIL / NOT-RUN per case | NOT-RUN, or a build older than the header → **stale-result** |
| **Result build** | The build each result was produced against | Older than the header build → **stale-result** (untrustworthy green) |
| **Evidence proves req?** | YES / WEAK / NO — does the captured proof make the requirement's claim true | WEAK/NO on a PASS → **contradiction / weak-evidence** |
| **Verdict** | `trustworthy-green` / a gap-class tag | The roll-up cell; only `trustworthy-green` counts as covered |

A separate **orphan list** sits below the grid — cases that trace to **no**
requirement can't appear as a row cell, so they're enumerated on their own:

```markdown
## Orphan cases (trace to no requirement)
| Case ID | Title | Likely cause | Action |
|---|---|---|---|
| TC-091 | "Legacy export to CSV" | Requirement retired in v3 | Retire case or relink |
```

## Verdict logic — coverage is a conjunction

A requirement is `trustworthy-green` only when **all three** hold:

```
covered = (a case traces to it)            ← else uncovered-requirement
        ∧ (fresh result against header build) ← else stale-result
        ∧ (evidence proves the claim)       ← else weak-evidence
```

Any leg missing flips the verdict from `trustworthy-green` to the corresponding
gap-class tag. The verdict column is therefore never "PASS" on its own — a raw
PASS with a stale build or weak evidence is *not* coverage.

## Worked example — all four gap classes

Build under audit: `v3.2.0-rc1`. The requirement set was triaged in Phase 0;
the cases and results were ingested from the manual run report.

| Req ID | Req prio | Requirement (testable form) | Case(s) | Result | Result build | Evidence proves req? | Verdict |
|---|---|---|---|---|---|---|---|
| REQ-101 | p0 | Checkout rejects an expired card with error "Card expired" | TC-010 | PASS | v3.2.0-rc1 | YES (screenshot of error) | **trustworthy-green** |
| REQ-102 | p0 | Order total includes tax for EU addresses | — | — | — | — | **uncovered-requirement (p0)** |
| REQ-103 | p1 | Password reset email arrives within 60s | TC-031 | PASS | v3.1.4 | YES | **stale-result (p1)** |
| REQ-104 | p1 | Cart persists across login | TC-042 | PASS | v3.2.0-rc1 | WEAK (asserts cart icon count, not item identity) | **weak-evidence (p1)** |
| REQ-105 | p2 | Footer shows current year | TC-055 | PASS | v3.2.0-rc1 | YES | **trustworthy-green** |

Below the grid:

```markdown
## Orphan cases (trace to no requirement)
| Case ID | Title | Likely cause | Action |
|---|---|---|---|
| TC-091 | "Bulk discount > 50 items" | Feature cut from v3 scope | Retire — route to PM |
```

### What the example surfaces

- **REQ-102 — uncovered requirement (p0).** A critical requirement (EU tax on
  the order total) with no case at all. Highest-stakes: nobody is testing it.
  Route to `qa-engineer` to author a case.
- **REQ-103 — stale result (p1).** TC-031 passed, but against `v3.1.4`, not the
  `v3.2.0-rc1` under audit. The green is untrustworthy — the email path may have
  changed since. Route to re-execute against the current build.
- **REQ-104 — weak evidence (p1).** TC-042 "passes", but its evidence only
  checks the cart *count*, not that the *same items* persisted — the requirement
  says cart *persists*, and identity isn't proven. A pass that doesn't prove its
  requirement. Route to strengthen the case (assert item identity) and re-run.
- **TC-091 — orphan case (p2).** Tests a bulk-discount feature that was cut from
  v3 scope. Effort spent on a requirement that no longer exists. Route to retire
  or relink.
- **REQ-101, REQ-105 — trustworthy-green.** A case traces, the result is fresh
  against `v3.2.0-rc1`, and the evidence proves the claim. These — and only
  these — count as covered.

So of five requirements, **two** are genuinely covered. A naive "5 cases, 4
pass" count would have reported 80% green; the matrix shows the real
trustworthy-green coverage is **2/5**, with one p0 hole, one stale critical
result, and one pass that doesn't prove its requirement. That delta is the
entire reason the matrix exists.

## Importing matrices from Excel

Requirement matrices and case lists often arrive as `.xlsx`. Convert with
[`xlsx-reader`](../../xlsx-reader/SKILL.md) before triangulating:

```bash
node scripts/read_xlsx.js requirements.xlsx /tmp/reqs.md
node scripts/read_xlsx.js cases.xlsx /tmp/cases.md
```

Read the produced Markdown back so each requirement and case becomes an
addressable row, then map them into the grid above. Keep the original column
that carries the case → requirement link — that link (or its absence) is what
distinguishes a covered requirement from an uncovered one and a real case from
an orphan.
